# 跨域调用与领域消息

跨域协作只有两条合法通道：**同步 `*-api`** 与 **异步领域消息（事件）**。禁止跨域直查他域表、禁止 `*-biz → *-biz`（见 [`architecture.md`](architecture.md)）。

本文约定：**何时同步 / 何时异步**、消息契约，以及传输层（当前 Redis、目标独立 MQ）。

## 1. 同步 vs 异步：评判标准

默认优先同步；满足下列条件才异步。

### 必须用同步 `*-api`

| 条件 | 说明 |
|------|------|
| 调用方需要返回值才能继续 | 如「按 userId 查昵称」「校验成员是否在租户内」 |
| 失败必须立刻反映给当前请求 | 如创建群失败、审批提交校验失败 |
| 与当前写操作同属一个业务事务语义 | 「做完 A 且 B 成功才算成功」且不能事后补偿 |
| 纯查询、幂等读 | 读通讯录、读配置等 |

### 应当用异步领域消息

| 条件 | 说明 |
|------|------|
| 一对多副作用 | 审批通过 → 通知收件箱 +（将来）日历/考勤等，生产方不应枚举所有下游 |
| 不阻塞用户主路径 | 写完本域主数据即可返回；下游可短暂延迟 |
| 可接受最终一致 | 下游晚到几秒、可重试，业务仍正确 |
| 生产方不应依赖消费方实现 | 解耦演进：新增消费者不必改生产者 |

### 禁止

| 反模式 | 原因 |
|--------|------|
| 「方便」就把查询改成异步再等回调 | 请求链路变复杂，超时难控 |
| 把强一致写操作拆成「写本地 + 发事件」却无补偿/对账 | 易出现半成功且无法感知 |
| 同一次用户请求里同步串联 3+ 域副作用 | 应用异步扇出或拆产品流程 |
| 用 WebSocket / Redis Pub/Sub 扇出冒充领域消息总线 | 那是实时投递，不是跨域业务事件总线（见 §3） |
| 为「看起来解耦」把本域内部两层 Service 也改成发消息 | 域内用方法调用即可 |

### 决策速查

```text
调用方是否必须在本 HTTP/WS 请求内拿到结果或失败？
  ├─ 是 → 同步 *-api
  └─ 否 → 是否有多家下游要听、且延迟可接受？
        ├─ 是 → 异步领域消息
        └─ 否 → 仍优先同步 *-api（简单可控）
```

**第一方「产品级」串联**（如审批通过默认进通知）也必须走上述通道，可以配置厚、逻辑多，但**仍禁止**跨域 `biz` 互依赖或直查他域表。

## 2. 消息契约

### Envelope（逻辑字段）

| 字段 | 要求 |
|------|------|
| `eventId` | 全局唯一（雪花或 UUID），消费幂等主键 |
| `eventType` | 稳定字符串，如 `bpm.process.approved` |
| `occurredAt` | 事件发生时间（UTC） |
| `tenantId` | 必填；与业务数据同租户 |
| `producer` | 域标识：`system` / `im` / `bpm` / … |
| `payload` | 仅 **DTO 形状**（可放在产出域 `*-api`）；禁止塞 DO / 他域内部类 |
| `schemaVersion` | 从 `1` 起；破坏性变更递增，旧消费者需兼容或并行双写期 |

### 发布时机

1. **本域业务写库成功之后**再发布（至少保证「本地已提交」）。
2. V1 可先「事务提交后发消息」；若出现丢消息不可接受，再引入 **Transactional Outbox**（表 + 转发），不改变 `eventType` / payload 契约。
3. 生产者只描述**已发生事实**（`*.approved`、`*.created`），不描述「请某下游做某事」的命令（命令优先同步 `*-api`）。

### 消费要求

- 消费者落在订阅域 `*-biz`，通过框架提供的订阅 API 注册；**不得**反射调用他域 Service。
- **幂等**：同一 `eventId` 重复投递不得产生重复业务副作用（去重表或业务唯一键）。
- **失败**：可重试错误进重试；不可恢复进死信/告警，禁止静默吞掉。
- 消费逻辑失败不得要求生产者回滚已提交事务（最终一致 + 补偿）。

## 3. 传输层：当前 Redis，目标独立 MQ

| 阶段 | 实现 | 说明 |
|------|------|------|
| **目标态（最终）** | 独立消息队列（如 RabbitMQ / RocketMQ / Kafka，立项时再定） | 持久化、消费组、死信、可运维监控；跨进程与 Phase 2 多 `*-server` 共用 |
| **当前（快速开发）** | **Redis 充当消息队列** | 不引入额外中间件；契约与 API 按最终 MQ 形状设计，便于替换 |

### 当前 Redis 用法约定

- 使用 **可排队、可消费组、可 ack** 的能力（推荐 **Redis Streams** + consumer group），**不要**用「发完即忘」的 Pub/Sub 做领域消息主路径。
- Key / Stream 名带租户或全局约定前缀，并与现有缓存 key 隔离，例如：`t:{tenantId}:de:{topic}` 或全局 `de:{topic}`（团队实现时统一一种，文档与代码一致）。
- 封装在 `relayflow-framework`（或 infra）的 **DomainEventPublisher / DomainEventListener** 背后；业务代码只依赖抽象，**禁止**业务里散落 `XADD` / 裸 Redis 命令。

### 与现有 Redis 用途区分

| 用途 | 机制 | 是否领域消息 |
|------|------|----------------|
| 缓存、JWT 黑名单、限流 | String / Hash 等 | 否 |
| WebSocket 多实例扇出 | Redis Pub/Sub（已有 `t:{tenantId}:ws:fanout`） | **否**（实时传输） |
| 跨域业务事件 | Redis Streams（当前）→ 独立 MQ（目标） | **是** |

替换 MQ 时：只换传输适配器与运维配置，**不改** `eventType`、payload DTO、同步/异步评判标准。

## 4. 与 `*-api` 的关系

```text
跨域读 / 要即时成败     → 同步调用对方 *-api
跨域广播已发生事实     → 发布领域消息（payload 类型可定义在产出域 *-api）
第三方将来接入         → 开放平台投影公开 HTTP/事件；内部仍用本套契约
```

开放平台仅在架构文档中占位，不在本文展开。
